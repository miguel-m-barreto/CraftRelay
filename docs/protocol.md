# Protocol

Sprint 4 adds the `craftrelay-journal` crate — the production durable local journal. It does not change v1 Protobuf contracts. The journal uses SQLite WAL mode with `synchronous=FULL`, `busy_timeout=5000`, `wal_autocheckpoint=1000`, and `foreign_keys=ON`. Schema version 1 includes tables: `journal_meta` (migration version), `stored_envelope` (immutable), `payload_blob`, `lifecycle_state` (revisioned with CAS), `dedup_fingerprint`, `producer_sequence` (conflict detection), `stream_head` (per-stream monotonic), `write_attempt` (bounded audit). The atomic local acceptance transaction allocates journal_sequence and stream_sequence, writes envelope + payload + lifecycle + dedup + producer sequence in one transaction, and returns LOCAL_ACCEPTED with delivery_status=LocalAccepted. Duplicate retry (same event_id + same fingerprint) returns ExistingDuplicate. Different fingerprint returns FingerprintConflict. Producer sequence conflicts are detected. Lifecycle uses CAS revisions — stale updates are rejected. Status recovery works after close/reopen. Disk guard rejects before transaction when unsafe. Shutdown rejects new writes. Corruption (tampered payload or lifecycle checksum) is detected explicitly. The journal never claims REPLICATED_DURABLE — Kafka delivery remains out of scope. The benchmark crate remains separate and benchmark-only.

Sprint 3 does not change the v1 Protobuf contracts. It adds the `craftrelay-journal-bench` crate as a benchmark-only spike for evaluating SQLite journal candidates (monolithic vs segmented). The benchmark transaction shape models the future Sprint 4 local acceptance transaction: journal sequence allocation, stream sequence allocation, immutable envelope record, payload blob record, and initial lifecycle state — all within one SQLite transaction. WAL mode with `synchronous=FULL` and configurable checkpoint/busy-timeout PRAGMAs is the baseline. Group commit splits transactions by both max-records-per-tx and max-bytes-per-tx. A real bounded writer pipeline (producer thread → sync_channel → dedicated writer thread → SQLite) produces real committed/failed transaction counts. Storage metrics include `db_size_bytes`, `wal_size_bytes`, `shm_size_bytes`, and `total_storage_bytes` across all DB/WAL/SHM files. Segmented candidate supports deterministic rollover by record count. Each CLI run uses an isolated data directory derived from the output path stem, cleaned before each run. No Protobuf changes, no DurableReceipt, no production journal API. The production journal backend decision remains EXPERIMENTAL pending Sprint 4.

Sprint 2 extends the v1 Protobuf contracts with producer registration (lifecycle states, priority/quota classes, registered handles, rejection codes), extended IntegrationManifest validation (event/query/namespace declarations, durability/retention/priority/quota classes, schema version ranges, validation codes including BEST_EFFORT rejection and self-promotion rejection), credential/ACL models (credential kinds/statuses, ACL principals/rules/scopes, allow/deny decisions with typed reason codes), policy resolution (admission decisions, rejection reasons including durability weakening/retention weakening/projection bypass/self-promotion/installation escape, effective policy results with policy version and ownership snapshot ID), NODE_LOCAL ownership snapshot (snapshot IDs/versions, namespace entries, ownership validation codes), quota/admission control (per-producer/namespace/global quota states, admission rejection reasons including P0 reserved capacity protection), and profile validation skeletons (Kafka durability profiles preserving RF=5/minISR=5/acks=all for P0, retention/projection/quota profiles, profile validation codes).

Sprint 1 adds bounded handshake, envelope-input, publish-status, typed-query freshness, barrier, authenticated token, and detachable watch request/response shapes. Java validates immutable client intent before transport submission. Installation and producer identity remain authenticated session inputs rather than arbitrary request selectors. The in-memory Agent and Query fixtures are explicitly non-durable and non-authoritative.

The v1 Protobuf skeleton is under `proto/craftrelay/v1`. Positive signed fields use `int32`/`int64`; zero is valid only for Kafka offsets. Repeated collections are policy-bounded. Core queries use typed `oneof` parameters or registered typed schema handles—never SQL or arbitrary schema names. Mutable fields and payload bytes are deliberately absent from `StoredEventEnvelope`.
